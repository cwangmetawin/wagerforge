## 1. Executive Summary — top changes and why

1. **`math-crash-family` (HIGH) — fix the inverse-CDF sampler.** Step 5 pairs the `u<h → bust at 1.0` atom with `crashPoint=(1−h)/u`, but the shipped Chase code and the RGS use `(1−h)/(1−u)` / `1/(1−u)`. The skill's pairing is internally inconsistent (emits sub-1.0 values near `u→1`, discontinuous at `u=h`). This is math a reader will copy verbatim into a money path. Also: `h` is per-game and server-sourced (Chase 0.02, Limbo 0.03), never one hardcoded constant; the N-rung note wrongly says each rung stakes `bet/N` when real code stakes full bet per tier.

2. **`build-host-bridge` (HIGH) — the slots path is materially wrong.** It claims a postMessage connector "brokers balance, bet, and spin" and "the parent shell debits, calls the RGS, and posts the result back." Reality: the slot game calls the RGS **directly** via the global slotify `window.connector` SDK (loaded as a separate non-bundled script); postMessage is a **one-way** UI/analytics notification (`spinStart{bet}`/`spinEnd{wonAmount}`, target `'*'`, never read back). The "validate origin of inbound messages" guidance is misattributed — the slot game never consumes inbound messages.

3. **Default-stack reality across many skills.** Mini-games are thin **React 19 + Vite over `@mini-games/studio`** (Zustand 5, TanStack Query 5, decimal.js) in a turbo/pnpm monorepo — Pixi/Phaser is the SLOT stack, not the minigame default (`build-minigame-from-scratch`). Money math is **not** one universal decimal.js wrapper: RGS uses `Number(value.toFixed(2|8))` + integer-scaling dust correction, host uses **bignumber.js v10**, decimal.js is client-studio-only with **inverted** roundUp/roundDown (`build-wallet-and-money`). Deploy target is **GKE**, not Cloud Run (`build-deploy-and-rtp-variant`). Mobile VRAM lever is **WebP atlas size-capping + Spine→sprite swaps**, not Basis/KTX2 which no game ships (`build-mobile-performance`).

4. **`fair-hash-chain` (MEDIUM) — verification is a server round-trip.** The real verifier posts `/game/proveFairness` with `{hashIndex, hash, seed, data:{rtp}}` and renders `randomizations[].gameEvent`; it does **not** recompute the chain client-side, and the multiplayer game set is exactly `crash0/slide/aviator/aviator0/sicbo` (not "crash/aviator"). `rtp` is a required proof input for crash-family.

5. **`build-game-server-rgs` (MEDIUM) — two corrections.** There is no `recover()` method on the RGS (`grep recover(` = 0 hits); reconnection is `replay(state, playerId)`. The now-live `IMultiplayerGame` tick model (phases, per-command accept-or-cancel idempotency, full-refund error recovery, deferred `pendingPayouts`) is entirely absent from the skill.

6. **Coverage gap: parimutuel / prediction-market game servers.** `snakes-pro/horse-pro/football-pro` ship continuous pool-proportional pricing, a secondary order book, per-buy fee, and `valuePerShare` settlement with dust-to-last-winner. `econ-tournament` owns the leaderboard/parimutuel altitude (extend it), but the **server implementation** pattern has no home → propose `build-prediction-market-games`.

7. **Three findings were adversarially REJECTED** (Section 6). Do not edit `build-slot-reels-symbols`'s curve-naming, do not add a cluster-grid section to `math-cascade-tumble` (owned by `math-lines-ways-cluster`), and do not treat the TS slot framework as "new since baseline" in `math-paytable-config`.
