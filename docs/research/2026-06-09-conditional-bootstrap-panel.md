## Consensus

- FAIL-OPEN IS IN THE WRONG PLACE AND CONFLATES TWO STATES. The spec (§3:51, §4:56) collapses 'genuinely absent superpowers' and 'settings present but unparseable/unreadable' onto a single fail-open-to-full. I verified live (Node v24.16.0) that JSON.parse throws on both `// comments` and trailing commas. A dual-install user with one hand-edited JSONC settings file therefore silently parses to 'not enabled' -> mode=full -> wagerforge AND superpowers both inject -> the exact double bootstrap the design exists to kill, on exactly the cohort it protects, with zero error/log/failing test. All 10 experts converge on splitting detection into a tri-state {matched->defer | absent->full | unreadable->full-but-surfaced}, isolating parse errors PER FILE so one bad scope cannot erase a good signal from another, and using strict `=== true` matching.
- THE HOOK'S MODE->EMITTED-JSON CONTRACT IS UNTESTED AND STRUCTURALLY UNTESTABLE UNDER THE DECLARED GATE. Verified: the gate `node --test 'scripts/**/*.test.mjs'` runs 82 tests and CANNOT see a hooks/-located test (a probe there is invisible); the brace form `{scripts,hooks}/**/*.test.mjs` runs 83 and picks it up. §6 only tests the pure functions; resolveBootstrapMode returning the STRING 'defer' does not prove the hook emits defer-only additionalContext. §9.1-9.4 are prose, not enforceable. Fix: extract a pure renderContext(mode, deps) into a scripts/-covered module, add positive full/defer output tests PLUS the negative anti-double-injection assertion (defer output must NOT contain '<EXTREMELY_IMPORTANT>' or 'using-superpowers'), and either co-locate in scripts/ or widen the gate glob and state the chosen glob in §6/§9.
- §6 LINE 83 CODIFIES THE BUG AS DESIRED BEHAVIOR. The test plan asserts 'malformed JSON -> false (fail-open)' as correct. For the dual-install case this rubber-stamps the double-injection regression. The malformed-WITH-superpowers-present case must assert defer (or at minimum a surfaced 'unreadable'), not silent false.
- DEFER IS A CROSS-HOOK DEPENDENCY ON AN UNOBSERVABLE SIBLING. `enabledPlugins.superpowers=true` means CONFIGURED, never RAN-THIS-SESSION; there is no shared result channel between plugins' SessionStart hooks (verified: superpowers even runs via a run-hook.cmd indirection layer, a larger failure surface). If superpowers is enabled but its hook is absent/evicted/errors, defer emits only the ~150-char nudge and NEITHER hook injects discipline. Consensus fix: make DEFER SELF-SUFFICIENT — its nudge carries a one-line standalone discipline pointer (e.g. 'before acting, check for a relevant skill via the Skill tool') so the worst case degrades to mild redundancy (which §1 already accepts) instead of silent zero-discipline.
- DETECTION MUST MATCH SUPERPOWERS UNDER ANY MARKETPLACE VIA THE /^superpowers@/ PREFIX, NOT AN EXACT KEY. Verified: a `superpowers-inline` data dir exists on this machine alongside the official one, proving non-official installs (`superpowers@inline`) are real. The spec's prefix RULE (§3:49) is correct; §3 prose and the entire §6 test plan only ever exercise the official key. Keep the prefix match and add §6 cases for `superpowers@inline` and `superpowers@<localname>`.
- HOME-DIR SOURCE MUST BE PINNED TO os.homedir(), NOT process.env.HOME. §3:48/§4:56 leave homeDir undefined at the real call site. An empty/overridden HOME silently mis-resolves the user-scope path -> superpowers key never found -> fail-open full -> double injection. Pass homeDir: os.homedir() explicitly and lock with a unit test using a non-HOME homeDir.
- THE ROUTER NUDGE MUST BE A SINGLE SHARED CONSTANT, NOT THREE HAND-WRITTEN COPIES. Plugin Law 1 (skill is single source of truth) is violated by the existing full-mode router string, the new defer nudge, and the using-wagerforge description drifting. Define one ROUTER_NUDGE constant reused in both modes; full mode = wrapper + nudge. Add a test asserting full-mode output literally contains the exact defer-mode nudge substring. The repo already proves this pattern (session-start.mjs builds `router` once and reuses it).
- VERSION-PARITY IS UNGUARDED. Verified: plugin.json and marketplace.json both say 0.2.0, validate.mjs references neither, and `git tag -l` is empty with no CHANGELOG. A two-file manual bump can drift and still pass the §9 done-gate. Add a tiny scripts/*.test.mjs asserting plugin.json.version === marketplace.json plugins[0].version, under the existing gate.
- DETECTION READS SETTINGS PATHS THAT ARE LOWER-TRUST THAN THE USER SCOPE. The three merged scopes have different trust levels; project/local files are clone-authored. Per-file try/catch isolation (everyone agrees) plus strict boolean matching are the cheap, high-value hardening; the heavier symlink/FIFO/size guards are contested (see unresolved).

## Unresolved disagreements

### Symlink/FIFO/size-cap DoS hardening of settings reads (lstat + regular-file check + 256 KiB cap)

**Positions:** security (Round 1) rated unbounded readFileSync of attacker-controllable project settings a MAJOR DoS/hang vector given async:false on the hook; failure-modes amplified it (async:false puts the read on the session-creation critical path). BUT security itself walked this back in Round 2 to a defense-in-depth MINOR (no privilege boundary crossed — the user reads their own files on their own machine). cc-platform and qa both argued against the symlink/lstat ceremony as defending a boundary that doesn't exist here, multiplying code paths the fail-open must traverse. All sides agree the cheap half (per-file try/catch + strict boolean) is worth doing.

**Chair recommendation:** Adopt the cheap half (per-file try/catch isolation + strict `=== true`) as a MUST — it is unanimous and also closes type-confusion. Treat the symlink/FIFO/size-cap guards as SHOULD-but-not-blocking: add a single statSync().size check before reading (a few lines, bounds the JSONC-bomb / accidental-huge-file case) and skip non-regular files, but do NOT build elaborate per-OS lstat traversal logic. security's own downgrade plus cc-platform/qa's objection make the heavy version over-scoped for a self-machine threat model; spend the complexity budget on the JSONC tri-state instead.

### Project-committable mode + a third 'lead' mode (read a wagerforge.bootstrap key from settings scopes; add WAGERFORGE_BOOTSTRAP=lead)

**Positions:** product argued the design's real target — the dual-install user who installed wagerforge FOR its iGaming routing — gets the WORST routing by default (defer hands discipline to superpowers, whose using-superpowers hard-routes 'brainstorm' to the bare skill with zero awareness of wagerforge domain prefixes), and that teams sharing project settings get per-developer-inconsistent modes with no committable pin. cc-platform, failure-modes, security, dx, release-compat, architecture, maintainability, qa, and red-team ALL pushed back: 'lead' is a near-synonym of force (zero new behavior, more surface); a 'lead' mode that keeps wagerforge's full <EXTREMELY_IMPORTANT> WHILE superpowers also fires literally re-enables the double injection as a feature; a new committable settings key deepens the documented-adjacent schema coupling that is the top maintainability risk; and it is a MINOR feature (0.3.0), not a 0.2.1 patch. product itself walked back the 'lead' alias in Round 2.

**Chair recommendation:** Reject 'lead' and the committable settings key for THIS release. The legitimate kernel (dual-install user wants wagerforge routing to win) is fully served by (a) WAGERFORGE_BOOTSTRAP=force, and (b) making the defer nudge DIRECTIVE — explicitly stating wagerforge:* skills take routing precedence for iGaming/slot/crypto tasks and that the duplicate skill entries are the pinned copies to prefer. Add product's strongest valid point as a SHOULD: a success criterion that tests ROUTING OUTCOME (defer mode routes an iGaming task into wagerforge:using-wagerforge), not merely nudge brevity. Defer the committable-mode/team-pin question to a follow-up 0.3.0 spec with its own behavior matrix; note it as a named future item so it isn't lost.

### Managed/enterprise settings scope (a 4th, highest-precedence layer above project-local: macOS /Library/Application Support/ClaudeCode/managed-settings.json etc.)

**Positions:** cc-platform (Round 1) rated it MAJOR: an enterprise admin force-enabling superpowers org-wide via managed settings is invisible to a 3-scope detector -> double injection for every managed user. release-compat, qa, and maintainability all argued NOT to block 0.2.1 on it: the file is absent on this machine, adding three more per-OS hardcoded paths widens the documented-adjacent-schema surface and adds untested branches to a fail-open module right before a patch ship, and the hand-editing-JSONC cohort dwarfs the managed cohort today.

**Chair recommendation:** Do NOT add the managed scope to 0.2.1. Architecture's seam fix makes this nearly free LATER: inject the settings-path LIST into the detector (not hardcoded paths), so adding the managed scope becomes a caller change, not detection-logic change. For now, name it as an explicit Non-goal in §7 with the stated consequence (managed-enabled superpowers -> double injection) so the gap is documented rather than silent. Ship the JSONC fix to the majority cohort first.

### Add 'resume' to wagerforge's SessionStart matcher

**Positions:** cc-platform/failure-modes/qa confirmed (verified) both wagerforge and superpowers use the identical `startup|clear|compact` (no resume) in 5.1.0 AND the commit-pin, so on a RESUMED session neither plugin injects and a standalone full-mode user silently loses the <EXTREMELY_IMPORTANT> discipline — counter to the design's 'never silently lose discipline' premise. dx warned that adding 'resume' to wagerforge alone (superpowers omits it) means a resumed dual-install user gets the wagerforge nudge with NO discipline above it — UNLESS defer is self-sufficient.

**Chair recommendation:** Decide it explicitly in §9 rather than leaving it unstated. Preferred: since the panel is already mandating a SELF-SUFFICIENT defer nudge (one-line discipline pointer), adding 'resume' to wagerforge's matcher becomes safe and is a strict improvement (standalone users keep discipline on resume; resumed dual-install users get the self-sufficient nudge, no double injection because superpowers stays silent on resume). If the owner prefers matcher-parity with superpowers, document resume as an accepted no-injection path with a success criterion noting the scope. Either way, the resume decision is gated on the self-sufficient-defer refinement landing first.

## Chair report

# Conditional-Bootstrap Design — Panel Verdict

## Verdict: REWORK SPEC FIRST
The approach (auto-detect superpowers + env override + defer) is sound and unanimously endorsed in shape. But the spec ships a detection path that **silently reintroduces the exact double-injection bug it exists to kill**, on exactly the cohort it protects, and its load-bearing deliverable is **untested and unrunnable under the declared gate**. Two of ten experts moved to needs-rework in Round 2 (qa, red-team, failure-modes); the rest are approve-with-refinements but agree the must-fixes are pre-implementation. These are contained, well-understood fixes — not an approach change — so rework is fast.

## The 5 things that matter most (all independently verified on this machine, Node v24.16.0)
1. **JSONC silently re-arms double injection.** `JSON.parse` throws on `// comments` AND trailing commas. The spec's single fail-open (§3:51) treats that throw as "superpowers not enabled" -> mode=`full` -> both plugins inject. A dual-install user with one hand-edited settings file gets the 2x bootstrap, no error, no log, no failing test. **Worse, §6 line 83 codifies this as desired behavior.**
2. **The hook contract is untested and untestable as specified.** `node --test 'scripts/**/*.test.mjs'` runs 82 tests and **cannot see a hooks/-located test** (I probed it); the brace form `{scripts,hooks}/**` runs 83 and does. §6 only tests pure functions; the mode->emitted-JSON wiring (where double injection actually occurs) has zero tests, so §9.1–9.4 are unenforceable prose.
3. **Defer trusts an unobservable sibling.** `enabledPlugins=true` means CONFIGURED, never RAN-THIS-SESSION (no cross-hook result channel; superpowers even runs via a `run-hook.cmd` indirection). If superpowers' hook is absent/evicted/errors, defer emits only the ~150-char nudge and **nobody injects discipline** — silent zero-discipline, the very failure fail-open exists to prevent.
4. **The fail-open is in the wrong place, and detection must not over-narrow.** Any throw in the hook body (e.g. a future packaging change breaking the new `hooks/`->`scripts/` import) bypasses the detector-internal fail-open and emits nothing — strictly worse than duplication. Separately, a `superpowers-inline` data dir exists here, proving non-official-marketplace installs are real: keep the `/^superpowers@/` prefix match (an exact-key match would false-negative it).
5. **Release hygiene is hollow.** No git tags, no CHANGELOG, both manifests say 0.2.0, and `validate.mjs` checks neither version. The patch bump silently flips a hook contract for dual-install users with no upgrader-facing record.

## Must-fix before implementation
- **Tri-state detection** (matched->defer, absent->full, unreadable->full), **per-file try/catch** so one bad scope can't poison others, strict `=== true`. Rewrite §6:83 so malformed-WITH-superpowers asserts defer.
- **Top-level try/catch in the hook** that emits the FULL bootstrap on ANY throw (import/fs/parse). Fail-open belongs at the I/O boundary, not only inside the detector.
- **Extract a pure `renderContext(mode)`**; add positive full/defer output tests + the **negative anti-double-injection assertion** (defer output must NOT contain `<EXTREMELY_IMPORTANT>` / `using-superpowers`). Co-locate in `scripts/` or widen the gate glob and state it in §6/§9.
- **Make defer self-sufficient**: its nudge carries a one-line standalone discipline pointer so worst case is mild redundancy, not zero discipline.
- **Pin `homeDir: os.homedir()`** (not `process.env.HOME`); keep the `/^superpowers@/` prefix and add an `superpowers@inline` test.

## Should-fix
Single shared `ROUTER_NUDGE` constant (Plugin Law 1) made directive about wagerforge:* routing precedence; version-parity test under the existing gate; CHANGELOG + reframe §8 as "behavior-changing-but-compatible" + surface `WAGERFORGE_BOOTSTRAP` in-session; lightweight `statSync().size` guard; a routing-OUTCOME success criterion + golden full-mode snapshot.

## Key unresolved tradeoff
**Project-committable mode / a third "lead" mode** (product). Real pain — the dual-install user who installed wagerforge *for* its iGaming routing gets the weakest routing by default, and teams can't pin a shared mode. But 9 experts (and product itself, in Round 2) reject it for 0.2.1: "lead" is a synonym of `force`, a committable settings key deepens the schema coupling, and a "lead-while-superpowers-fires" mode literally re-enables the bug as a feature. **Recommendation:** serve the kernel now via `force` + a directive defer nudge + a routing-outcome test; defer committable team-mode to a 0.3.0 spec. **Managed/enterprise scope** and **adding `resume` to the matcher** are similarly real but secondary: name managed scope a §7 Non-goal with its consequence; once defer is self-sufficient, adding `resume` to wagerforge's matcher becomes a safe strict improvement.
